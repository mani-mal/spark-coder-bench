# Roster gap: Qwen3.6 family (27B dense / 35B-A3B) — and an NVFP4-native coder opportunity

**Date:** 2026-07-08
**Status:** Proposal for peer review — NOT yet a decision to add an arm.
**Scope:** Model-roster currency + a potential arm that touches the study's central
serving-feasibility finding.

---

## 1. What we missed

The study roster (gpt-oss-120b, Nemotron-3-Super-120B-A12B, Qwen3-Coder-30B-A3B-Instruct)
was fixed around the **older `Qwen3-Coder` generation**. Two newer Qwen models shipped
**before this project started (late June 2026)** and were never considered:

| Model | Released | Arch | Total / Active | SWE-bench Verified (vendor/3rd-party) | Official NVFP4? |
| --- | --- | --- | --- | --- | --- |
| Qwen3.6-35B-A3B | 2026-04-16 | MoE (256 exp, top-8 + shared) | 35.95B / ~3B | 73.4 (reported) | **yes — `nvidia/Qwen3.6-35B-A3B-NVFP4`** |
| Qwen3.6-27B (dense) | 2026-04-22 | Dense | 27.8B / 27.8B | 77.2 (reported) | yes — `nvidia/Qwen3.6-27B-NVFP4` |

Scores above are **vendor / third-party reported**, not independently measured — precisely
the kind of claim this harness exists to test.

## 2. Why 35B-A3B is the relevant one for *this box*

- **Same active-param speed class as our current Qwen arm** (~3B active vs Qwen3-Coder's
  3.3B), so on the bandwidth-bound GB10 (~273 GB/s LPDDR5x) it decodes in the same tier —
  no latency regression — while being a full generation newer on coding/agentic tasks.
- **Fits the disclosed sparsity spectrum** cleanly: 35.95B/3B ≈ 8.3% active, sitting next
  to Qwen3-Coder (10.8%), gpt-oss (4.4%), Nemotron (10.0%).
- The 27B **dense** model is a quality ceiling but all-27.8B-active → ~10–15 tok/s on this
  box; interesting as a "dense vs MoE at equal quality" contrast, not as a daily driver.

## 3. The methodological hook (the real reason to raise it)

Our capstone serving-feasibility result is: **under TRT-LLM 1.3.0rc9 on sm_121a, only the
FP4-native NVIDIA reasoning model (Nemotron) serves; community MoE *coders* run on vLLM
only** (qwen bf16 MoE + gpt-oss MXFP4 both blocked on TRT-LLM). The README explicitly notes
the design **cannot isolate a generic FP4-on-Blackwell effect** because no model serves on
both runtimes and the two 4-bit MoEs use different runtimes/quants.

`nvidia/Qwen3.6-35B-A3B-NVFP4` is an **NVFP4-native MoE _coder_ published by NVIDIA**. That
is exactly the missing arm:

- If it serves on **TRT-LLM**, we'd have a *coder* on the same runtime+quant family as
  Nemotron — moving toward the matched comparison the current design admits it lacks.
- If it also serves on **vLLM** (Marlin weight-only), it could be the first model that runs
  on **both** runtimes → the same-model vLLM↔TRT-LLM bridge we declared "not achievable"
  (see `2026-06-29-full-matrix-results.md` §1) might become achievable with this checkpoint.

Both are **empirical unknowns** — GB10/sm_121a has no native FP4 *compute*, and TRT-LLM
1.3.0rc9's MoE path rejected every community coder we tried. But an NVIDIA-published NVFP4
coder is the single most likely candidate to break that stalemate, so it is worth a serve
probe regardless of whether we add it as a scored arm.

## 4. Cost / effort

- Adding a scored arm = 1 entry in `infra/models.json` + a matching profile (per README),
  **but** a full re-run across L1 (SWE-bench subset) / L2 (app build) / L3 (LiveCodeBench)
  at held seeds/temperature is real compute and re-aggregation.
- A **serve-feasibility probe only** (does `nvidia/Qwen3.6-35B-A3B-NVFP4` load on TRT-LLM
  and/or vLLM on this box?) is cheap and, by itself, updates a primary finding.

## 5. Recommendation (pending peer review)

1. **Cheap first:** run a serve-feasibility probe of `nvidia/Qwen3.6-35B-A3B-NVFP4` on both
   runtimes. Outcome updates the serving-feasibility result either way.
2. **If it serves:** consider adding it as a current-gen Qwen coder arm (likely replacing or
   sitting beside Qwen3-Coder-30B-A3B as a legacy control), and — if it serves on both — use
   it for the previously-blocked same-model runtime bridge.
3. Treat Qwen3.6-27B dense as an optional dense-vs-MoE side contrast, not a core arm.

Framing stays intact: this remains a **systems + methodology study of deployment
configurations**, not a model-ranking paper. The addition is about roster currency and
closing a disclosed design gap, not chasing a leaderboard.
